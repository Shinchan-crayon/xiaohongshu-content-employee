# 个性化学习规则

## 目标

在不修改插件公共知识、不自动采信外部内容的前提下，让工作流持续适应用户、账号和品类。

## 四类信息

1. **公共知识**：插件内置的平台、内容、审核和视觉规则，只随插件版本更新。
2. **当前素材**：本次任务读取的产品资料、链接、图片和搜索结果，默认只服务当前任务。
3. **用户偏好**：用户明确表达、选择或修改产生的偏好，可以自动保存。
4. **候选知识**：外部事实、运营规律和内容模式，先保存为 `pending`，用户批准后才能长期复用。

## 启动引导

每个新任务开始时必须向用户展示：

> 个性化学习已开启：你明确修改和偏好会自动保存；外部事实和运营规律需要确认后才会长期使用；你可以随时查看学习记录、关闭个性化学习或删除学习记录。

启动后读取 `context`，并用大白话列出本次实际采用的历史偏好。没有可用记录时说明“本次暂未使用历史偏好”，不要假装已经了解用户。

## 自动保存条件

只在以下证据明确时调用 `record-preference`：

- 用户直接说“以后都这样”“我的账号不要这样写”等长期偏好。
- 用户明确选择固定语气、标题倾向、正文结构或视觉习惯。
- 用户实际修改内容，并明确说明修改原因。
- 同一偏好再次出现时更新观察次数，不创建重复记录。

沉默、模糊评价、模型推测和一次性的任务限制不得自动保存为用户偏好。

## 外部知识确认

调研中发现可跨任务复用的事实、运营规律或内容模式时：

1. 保留简短结论、来源和观察日期。
2. 调用 `propose` 保存为 `pending`。
3. 在任务结束前向用户展示候选内容。
4. 用户明确批准后调用 `review --decision approve`。
5. 用户拒绝时调用 `review --decision reject`。

`pending` 和 `rejected` 内容不得出现在后续任务的 `context` 中。搜索摘要、竞品原文和无法追溯来源的结论不得直接写入已批准知识。

## 用户控制

- “查看学习记录”：调用 `status` 和 `context`，展示偏好、已批准知识和待确认数量。
- “关闭个性化学习”：调用 `disable`；关闭后不读取也不新增个性化数据。
- “开启个性化学习”：调用 `enable`。
- “删除学习记录”：先展示记录 ID，再对用户指定项调用 `forget`。

## 存储边界

使用 `../../scripts/知识迭代工具/manage_knowledge.py` 管理数据。用户数据存放在插件目录之外；工作流状态只记录 `xhs-user-data://knowledge` 和记录 ID，不写入本机绝对路径。

不得保存密码、访问令牌、密钥、联系方式或大段受版权保护的原文。
